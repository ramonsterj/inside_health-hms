# i18n / Reference-Data Labels

The app's default locale is **Spanish**. Several user-facing strings derive from
stable backend **codes** (role codes, permission codes, warehouse codes, …) whose
DB `name`/`description` columns were historically seeded in **English**. This
document defines the project standard for rendering those labels so Spanish users
never see English, and EN/ES both work.

## The rule

**Any label derived from a stable backend code MUST be rendered through a
vue-i18n key derived from that code. The DB `name`/`description` is a *fallback
only* — used for admin-created rows not yet in the bundle.**

Codes covered by the rule: role code, role description, permission code,
permission *resource* (group header), document-type code, triage code, warehouse
code. Codes are stable and bilingual-ready; keying on them survives DB text edits
and supports EN/ES simultaneously.

### Seeded-code vs admin-free-text boundary

| Kind | How it's rendered | Stored in |
| ---- | ----------------- | --------- |
| Stable seeded code (role, permission, warehouse, document type, triage code) | **i18n key** (`useCodeLabels`), DB value as fallback | DB (English historically; V128 Spanishizes as defense-in-depth) |
| Admin-authored free-text **without** a stable code (inventory *item* names, lab test names, admin-renamable inventory/psychotherapy *categories*) | **directly from the DB** (no key to translate on) | DB, in Spanish |

The admin-free-text rows are explicitly *out* of the i18n-key rule — there is no
stable key to translate on. V128 fixes their seeded text to Spanish, but the UI
renders them straight from the DB.

## How to render — `useCodeLabels`

`web/src/composables/useCodeLabels.ts` is the single entry point. It wraps
`useI18n()` (stays locale-reactive) and returns typed helpers:

```ts
const { roleName, roleDescription, permissionName, permissionDescription,
        permissionGroupLabel, documentTypeName, triageCodeLabel,
        warehouseName, warehouseDescription } = useCodeLabels()

roleName('ADMINISTRADOR', role.name)              // roleNames.ADMINISTRADOR
roleDescription(role.code, role.description)       // roleDescriptions.<CODE>
permissionName(p.code, p.name)                     // permissions.<code>.name
permissionDescription(p.code, p.description)        // permissions.<code>.description
permissionGroupLabel(resource)                      // roles.permissionGroups.<resource>
documentTypeName(dt.code, dt.name)                  // document.types.<code>
triageCodeLabel(tc.code, tc.description)            // triageCode.codes.<code>
warehouseName(w.code, w.name)                       // warehouse.names.<CODE>
warehouseDescription(w.code, w.description)         // warehouse.descriptions.<CODE>
```

Each helper resolves `${namespace}.${code}` via `te()` and falls back to the
supplied DB value (then the raw code) when the key is absent.

> **NEVER** render `role.name`, `role.description`, `permission.name`,
> `permission.description`, a warehouse `name`/`description`, a document-type
> `name`, or a triage `description` raw in a template. Always go through
> `useCodeLabels`.

### Key naming map

| Concept | Key | Source code |
| ------- | --- | ----------- |
| Role name | `roleNames.<CODE>` | `roles.code` |
| Role description | `roleDescriptions.<CODE>` | `roles.code` |
| Permission name | `permissions.<code>.name` | `permissions.code` |
| Permission description | `permissions.<code>.description` | `permissions.code` |
| Permission group header | `roles.permissionGroups.<resource>` | distinct `permissions.resource` |
| Document type name | `document.types.<code>` | `document_types.code` |
| Document type description | `document.typeDescriptions.<code>` | `document_types.code` |
| Triage code | `triageCode.codes.<code>` | `triage_codes.code` |
| Warehouse name | `warehouse.names.<CODE>` | `warehouses.code` |
| Warehouse description | `warehouse.descriptions.<CODE>` | `warehouses.code` |

Keys may contain `:` (e.g. `permissions.user:create.name`). vue-i18n only splits
on `.`, so the `:` stays inside one path segment and resolves against the nested
`"permissions": { "user:create": { "name", "description" } }` object.

## The two guards

ESLint `no-raw-text` does **not** catch DB-field bindings like
`{{ permission.name }}`, so two automated guards enforce completeness instead:

1. **Backend coverage test** (`api/.../i18n/I18nReferenceDataCoverageTest.kt`) —
   after Flyway migrates the Testcontainers DB (the DB is the source of truth for
   codes; there is no Kotlin enum), it queries `roles.code`, `permissions.code`,
   distinct `permissions.resource`, `warehouses.code`, `document_types.code`, and
   `triage_codes.code`, then asserts the web `es.json` bundle has **exactly** the
   required key for each — no missing keys (a new code can't ship without a
   translation) and no stale keys.
2. **Frontend catalog + vitest** (`web/src/i18n/catalogs.ts` +
   `catalogs.spec.ts`) — `ROLE_CODES`, `PERMISSION_CODES`,
   `PERMISSION_RESOURCES`, `WAREHOUSE_CODES`, `DOCUMENT_TYPE_CODES`, `TRIAGE_CODES`,
   with a vitest asserting each entry has its required key in **both** locales and
   that the bundle has no keys beyond the catalog. Fast, no DB; doubles as documentation.

The existing `locales.parity.spec.ts` (EN ⇄ ES key symmetry + non-emptiness)
auto-covers the new sections. Together these close the **DB ↔ catalog ↔ i18n**
loop: backend proves bundle == DB, frontend proves catalog == bundle == both
locales.

## When you add a role / permission / warehouse

1. Add the row in a migration (e.g. `V129`).
2. Add its code to `web/src/i18n/catalogs.ts`.
3. Add its key(s) to **both** `es.json` and `en.json`.
4. Spanishize the DB text in the migration and the matching `R__seed_*` file,
   bumping `SEED-BUNDLE-VERSION` across all nine seed files (see
   `docs/architecture/MIGRATIONS.md`).

The guards fail until every code has its keys — so drift can't merge.
